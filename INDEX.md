# MakeOps Repository Index

This document provides a granular and normative map of all technical artifacts stored within the MakeOps repository. It explicitly defines the architectural role and engineering purpose of every project file, ensuring complete transparency and facilitating precise context management for both developers and automated systems.

## Project Management & Configuration (Root)

This section details the foundational files located in the project's root directory, which collectively govern its legal identity, build orchestration, and top-level management. These artifacts provide the essential bootstrapping logic and compliance frameworks necessary to maintain the integrity and reproducibility of the entire software suite.

- `.gitignore`: The Git version control exclusion file, defining the rules for ignoring ephemeral and environment-specific artifacts. It prevents the accidental commitment of build directories, local development settings, and sensitive secrets into the shared repository.
- `INDEX.md`: This file serves as the normative and comprehensive index of all technical artifacts within the MakeOps repository. It defines the architectural purpose of every project file to ensure total transparency and facilitate efficient context ingestion for both developers and automated systems.
- `LICENSE`: The normative legal document providing the full and exact text of the GNU General Public License version 3 (GPLv3). It defines the legal framework for the copying, distribution, and modification of the software, ensuring its status as Free Software.
- `Makefile`: The operational facade of the project, designed to encapsulate and streamline complex development workflows. It leverages GPRbuild and custom DevOps scripts to automate critical tasks such as building, formal verification, unit testing, and repository synchronization.
- `makeops.gpr`: The master GPR aggregate project file that serves as the central configuration point for the GPRbuild system. It logically integrates the main application project with the unit testing infrastructure to provide a unified and consistent build environment.
- `README.md`: The primary document outlining the MakeOps project, providing a comprehensive overview of its deterministic architecture and key features. It establishes the Specification-Driven Development (SDD) methodology and provides essential instructions for building and maintaining the software suite.
- `REUSE.toml`: The configuration file for the REUSE tool, ensuring the project adheres to strict licensing and copyright transparency standards. It formally declares the GPL-3.0-or-later license and copyright information for non-code artifacts such as `.gitignore` and `README.md`.

## DevOps Tooling & Automation

- `devops/scripts/check_docs_status.sh`: Skrypt weryfikujący, czy wszystkie dokumenty analityczne w podkatalogach `docs/specification/` posiadają status `APPROVED`. Egzekwuje on rygory metodyki SDD, zapobiegając commitowaniu szkiców (DRAFT) lub dokumentów przestarzałych (DEPRECATED).
- `devops/scripts/commit_ai.sh`: Skrypt automatyzujący proces dodawania commitów wspomaganych przez AI. Umożliwia wczytanie wygenerowanego opisu commita bezpośrednio ze standardowego wejścia (stdin) lub ze wskazanego pliku argumentem `--input-file`.
- `devops/scripts/commit_interactive.sh`: Interaktywny kreator tekstowy (CLI wizard) do ręcznego tworzenia commitów. Krok po kroku odpytuje użytkownika o kategorię, tytuł, opis, listę zmian (SCOPE OF WORK) i metadane, gwarantując pełną zgodność komunikatu z rygorystyczną konwencją projektu.
- `devops/scripts/ensure_license_headers.sh`: Idempotentny skrypt wymuszający obecność nagłówków licencji GPLv3 w plikach źródłowych (.ads, .adb, .gpr, .sh itp.) i wykonujący weryfikację narzędziem `reuse`.
- `devops/scripts/generate_staged_diff.sh`: Narzędzie pomocnicze generujące plik `changes.diff` na podstawie zmian oczekujących (staged) w repozytorium. Służy do dostarczania precyzyjnego kontekstu dla asystentów AI oraz ostrzega przed niezamierzonymi, częściowymi commitami.
- `devops/scripts/push_repo.sh`: Skrypt do synchronizacji repozytorium z GitHubem; weryfikuje zmiany i wymaga pliku `.secrets`.

## Documentation - Requirements Specifications (Phase 1)
Documents defining the fundamental Capabilities of the system and architectural constraints from the perspective of functional and non-functional requirements. In accordance with the Specification-Driven Development (SDD) methodology, they form the foundation for further analytical and design work.

- `docs/specification/REQ-000-system-constraints.md`: The baseline document establishing the global architectural, technological (Ada 2022, SPARK), and legal (GPLv3) frameworks for the entire MakeOps tool. It establishes the requirement of compiling into a single standalone `mko` binary.
- `docs/specification/REQ-001-project-configuration.md`: Specification of the Capability for users to define the operational environment. It describes the requirements for handling the `makeops.toml` file, including the declaration of constants (`[environment]`), operations (`[operations.<name>]`), and their mutual dependencies.
- `docs/specification/REQ-002-operation-orchestration.md`: Specification of the core MakeOps engine Capability: task orchestration and execution. It defines the process of dependency resolution (DAG), variable substitution, and safe, direct process invocation (Pure Execution) bypassing the system shell.
- `docs/specification/REQ-003-execution-observability.md`: Document describing the Capability of monitoring the tool's operation by the user via the CLI (`mko`). It defines three levels of log verbosity (Error, Normal, Debug), rules for handling child process streams (`stdout`/`stderr`), and standard exit codes.
- `docs/specification/REQ-004-global-preferences.md`: Specification of the Capability for global configuration of the `mko` tool. It introduces a 5-level settings override hierarchy (from defaults, through `/etc/makeops/config.toml` and `~/.config/makeops/config.toml` files, up to environment variables and CLI flags).

## Documentation - Theoretical Foundations and Conceptual Models (Phase 2)
Analytical documents exploring the "First Principles" of the system. They separate pure theoretical knowledge, mathematics, and strict platform constraints (POSIX, SPARK) from concrete code implementations. These models serve as the absolute substantive foundation for the architectural design space.

- `docs/specification/MOD-001-master-orchestration-lifecycle.md`: Defines the high-level, "Fail-Fast" operational flow of the system. It maps out the strict initialization, execution, and graceful degradation phases of the MakeOps tool.
- `docs/specification/MOD-002-universal-processing-pipeline.md`: Describes the Zero-AST processing architecture used for parsing and evaluating configurations without dynamic memory allocation. It splits data ingestion into a strict 5-phase pipeline (Lexer, Parser, Normalizer, Validator, Applier).
- `docs/specification/MOD-003-hierarchical-configuration-cascade.md`: Details the override priority model for merging configurations from multiple sources (defaults, global `/etc/` and `~/` files, project files, OS environment variables, and CLI flags) into a single, deterministic state.
- `docs/specification/MOD-004-execution-plan-resolution.md`: Explains the mathematical representation of operations as a Directed Acyclic Graph (DAG). It covers the Topological Sorting algorithm used to resolve execution order and the safe engine for Lazy Variable Substitution.
- `docs/specification/MOD-005-asynchronous-execution.md`: Focuses on the real-time event loop and process multiplexing. It details how the orchestrator uses non-blocking POSIX calls (`poll`) to monitor child processes and manage time-bound execution (Grace Periods) without blocking the main thread.
- `docs/specification/MOD-006-posix-ipc-stream-routing.md`: Details the Inter-Process Communication (IPC) architecture. It explains the use of POSIX anonymous pipes and file descriptors to securely route, multiplex, and isolate child process standard streams (`stdout`, `stderr`) from the parent orchestrator.
- `docs/specification/MOD-007-pure-execution-os-boundaries.md`: Establishes the "Pure Execution" principle, where child processes are invoked directly via the `execvp` system call without shell intermediaries. It focuses on C ABI compatibility, safe argument passing, and the complete elimination of shell-injection vulnerabilities.
- `docs/specification/MOD-008-system-signal-routing.md`: Defines how the orchestrator intercepts and routes POSIX signals (e.g., `SIGINT`, `SIGTERM`). It describes the conversion of asynchronous hardware interrupts into deterministic state flags within the SPARK core to ensure graceful and synchronized shutdowns.
- `docs/specification/MOD-009-formal-verification-static-memory.md`: Serves as the system's "Safety Constitution," establishing rules for SPARK Silver Level verification. It details the zero-heap memory model, utilizing package-level static allocation (BSS/Data) and bounded types to prove the Absence of Runtime Errors (AoRE).
- `docs/specification/MOD-010-text-encoding-byte-bucket.md`: Outlines the "Raw Byte Bucket" paradigm for handling UTF-8 text using Ada's 8-bit `String` type. It explains how to maintain full Unicode compatibility and POSIX interoperability without the memory overhead or verification complexity of `Wide_Wide_String`.
- `docs/specification/MOD-011-isolated-os-boundaries.md`: Describes the protective Facade layer between the core logic and the OS. It introduces the "Dual-Pragma" isolation strategy (On for `.ads`, Off for `.adb`) to trap unpredictable native exceptions and return deterministic, verified results.
- `docs/specification/MOD-012-execution-context-security-model.md`: Details workspace security and directory management. It defines the "Configuration Anchor" vs. "Execution Context" distinction and implements the "Opt-In Root" safety mechanism to prevent unintentional filesystem pollution.
- `docs/specification/MOD-013-toml-dialect-model.md`: Specifies the highly restricted, String-Only TOML dialect used for configuration. It describes the design of a custom finite-state machine (FSM) parser that achieves $O(1)$ memory complexity and eliminates the need for an Abstract Syntax Tree (AST).
- `docs/specification/MOD-014-env-variables-shadowing-model.md`: Defines the strict namespace separation between global orchestrator settings (`MKO_`) and project-level parameters (`OS_Env`). It establishes the immutable shadowing hierarchy and anti-injection rules for safe, non-recursive variable substitution.
- `docs/specification/MOD-015-cli-arg-normalization.md`: Describes the POSIX-compliant parsing of Command Line Interface (CLI) arguments. It details the finite-state machine (FSM) heuristics used to securely scan, normalize, and separate execution flags from positional target names.
- `docs/specification/MOD-016-observability-taxonomy.md`: Establishes the visual formatting rules for terminal output. It defines the "Neutral Happy Path" principle, standardized diagnostic prefixes (`[mko:*]`), strict ANSI color usage, and UTF-8 emoji markers to ensure a predictable and highly accessible Developer Experience (DX).
- `docs/specification/MOD-017-zero-alloc-diagnostics.md`: Details the mechanism for providing Contextual Error Rendering without dynamic memory allocation (Zero-AST). It introduces the "Hot Path vs. Cold Path" dichotomy and the Just-In-Time (JIT) file re-open technique to extract exact error context upon failure.
- `docs/specification/MOD-018-project-config-schema.md`: Defines the absolute, normative schema for the project-level `makeops.toml` file. It establishes the rules for defining the static Environment Dictionary and the Operations Graph (DAG) nodes, enforcing strict semantic validation and the "String-Only" paradigm.
- `docs/specification/MOD-019-global-prefs-schema.md`: Specifies the restricted structure of user and system-level configuration files (e.g., in `/etc/` or `~/.config/`). It enforces the Separation of Concerns and system security by strictly prohibiting execution logic (`[operations]`) and hardcoded privilege escalations (`allow_root`) in global scopes.

## Documentation - Architectural Package Designs (Phase 3)
Normative documents developed in accordance with the Specification-Driven Development (SDD) methodology, serving as the "glue" between Requirements (REQ), Conceptual Models (MOD), and the actual code. They define package responsibility boundaries, interface semantics, implementation guidelines (especially at the intersection of strict SPARK constraints and the operating system), and formal verification strategies.

**Base Packages (Cluster 0):**
- `docs/specification/DES-000-makeops.md`: Design of the absolute root namespace package for the MakeOps software suite. It establishes global project identity and static metadata while strictly enforcing the `pragma Pure` constraint to guarantee zero hidden state across all child subsystems.
- `docs/specification/DES-001-makeops-app.md`: Design of the root namespace for the Application Layer. It defines foundational types representing the global application state, specifically establishing the strict `Log_Level` enumeration (Error, Info, Debug) used to control diagnostic output granularity.
- `docs/specification/DES-002-makeops-core.md`: Design of the foundational layer for the core business logic and orchestration engine. It defines platform-independent domain types, enforcing a "Functional Core" architecture that replaces native Ada exceptions with deterministic, mathematically verifiable return types (e.g., `Operation_Result`).
- `docs/specification/DES-003-makeops-sys.md`: Design of the root namespace and foundational abstraction layer for all Operating System and hardware interactions. It introduces controlled boundaries for unrecoverable hardware exceptions (`System_Error`) and defines standard POSIX exit codes and system error mappings.
- `docs/specification/DES-004-makeops-tests.md`: Design of the root namespace and main aggregation point for the AUnit testing infrastructure. It defines the master test suite and provides the entry point for the standalone `test_runner` executable, operating outside the strict SPARK constraints.

**OS & Memory Foundations (Cluster 1):**
- `docs/specification/DES-005-makeops-sys-env.md`: Design of the safe OS Facade for querying host environment variables. It enforces Graceful Degradation by trapping native Ada exceptions and translating them into deterministic `Env_Result` variant records, strictly guaranteeing the Absence of Runtime Errors (AoRE).
- `docs/specification/DES-006-makeops-sys-fs.md`: Design of the exception-free file system adapter. It manages standard Linux path resolution (XDG Base Directory Specification), safe working directory transitions (`chdir`), and Pre-flight Executability Checks without propagating native I/O exceptions into the pure core.
- `docs/specification/DES-007-makeops-sys-fs-os_bindings.md`: Design of the private, unsafe "Thin Binding" layer to the Linux kernel and `glibc`. It explicitly opts out of SPARK verification (`SPARK_Mode (Off)`) to map raw POSIX C functions (e.g., `access`, `chdir`, `realpath`) to Ada equivalents for internal consumption by the `MakeOps.Sys.FS` wrapper.
- `docs/specification/DES-008-makeops-sys-processes.md`: Design of the safe "Thick Wrapper" for managing the lifecycle of POSIX processes. It provides strongly-typed, SPARK-safe abstractions for spawning processes (Pure Execution), monitoring streams via IPC pipes, and deterministically translating raw POSIX return codes into verified domain states.
- `docs/specification/DES-009-makeops-sys-processes-os_bindings.md`: Design of the private "Thin Binding" layer for process and Inter-Process Communication (IPC) management. It defines the unprovable C ABI interfaces for native Linux system calls (e.g., `fork`, `execvp`, `pipe`, `poll`) strictly isolated from the higher-level orchestration logic.
- `docs/specification/DES-010-makeops-sys-terminal.md`: Design of the safe, exception-free OS Facade for standard terminal streams (`stdout`, `stderr`). It isolates the verified core from native I/O library errors (e.g., broken pipes) by trapping and safely suppressing unprovable `Ada.Text_IO` exceptions, while enforcing automatic stream flushing for real-time observability.
- `docs/specification/DES-011-makeops-sys-signals.md`: Design of the thread-aware OS adapter for hardware interrupt routing. It encapsulates Ada concurrency mechanisms (protected objects) to safely and asynchronously intercept system signals (e.g., `SIGINT`, `SIGTERM`), converting them into deterministic state flags without blocking the main orchestration loop.
- `docs/specification/DES-012-makeops-sys-identity.md`: Design of the safe, SPARK-verifiable OS Facade for querying operating system user identity. It provides a deterministic mechanism to check for superuser (root) privileges, serving as the foundational safety check to prevent accidental Workspace Pollution.
- `docs/specification/DES-013-makeops-sys-identity-os_bindings.md`: Design of the private, unsafe "Thin Binding" layer to the Linux kernel and `glibc`. It explicitly opts out of SPARK verification to import the raw POSIX C function `getuid`, securely isolating it from the higher-level domain logic.
- `docs/specification/DES-014-makeops-sys-time.md`: Design of the safe, SPARK-friendly OS adapter for monotonic time and duration measurements. It guarantees deterministic, integer-based time interval calculations necessary for logging execution duration and managing non-blocking I/O Grace Period timeouts.
- `docs/specification/DES-015-makeops-sys-file_stream.md`: Design of the exception-free OS adapter for sequential, line-by-line text file reading. It encapsulates native `Ada.Text_IO` calls to prevent exception leakage (such as `End_Error`), translating them into deterministic variant records and bounded strings ready for safe consumption by the Zero-AST Lexer.

- `source/sys/makeops-sys-env.adb`: Implementacja fasady zmiennych środowiskowych, izolująca natywne wyjątki OS (`Constraint_Error`) i zapewniająca łagodną degradację.
- `source/sys/makeops-sys-file_stream.ads`: Specyfikacja bezpiecznej, wolnej od wyjątków fasady do strumieniowego odczytu plików tekstowych (linia po linii). Wykorzystuje ograniczone ciągi znaków (Bounded Strings) i rekordy wariantowe, gwarantując brak błędów w czasie wykonania (AoRE).
- `source/sys/makeops-sys-file_stream.adb`: Implementacja fasady strumieni plików. Hermetyzuje natywne wywołania `Ada.Text_IO`, zarządza mechanizmem ucinania zbyt długich linii (zapobieganie przepełnieniu bufora) i mapuje natywne wyjątki OS (np. `Name_Error`, `Status_Error`) na bezpieczne, deterministyczne stany odczytu.
- `source/sys/makeops-sys-fs-os_bindings.ads`: Cienkie wiązania (Thin Bindings) do standardowych funkcji POSIX systemu plików w C (np. `access`, `chdir`, `getcwd`, `realpath`).
- `source/sys/makeops-sys-fs.ads`: Specyfikacja bezpiecznej fasady do interakcji z systemem plików i sprawdzania uprawnień w oparciu o deterministyczne wyliczenia `FS_Status`.
- `source/sys/makeops-sys-fs.adb`: Implementacja fasady systemu plików, zarządzająca pamięcią na granicy Ada-C i chroniąca przed awariami (AoRE).
- `source/sys/makeops-sys-identity-os_bindings.ads`: Prywatny pakiet z "cienkim wiązaniem" (Thin Binding) do funkcji `getuid` z biblioteki glibc, mapujący natywne typy POSIX. Celowo wyłączony z rygoru weryfikacji SPARK.
- `source/sys/makeops-sys-identity.ads`: Specyfikacja rygorystycznej, w pełni weryfikowalnej w SPARK fasady tożsamości (Thick Wrapper). Wystawia pojedynczy, bezpieczny kontrakt `Is_Root_User`.
- `source/sys/makeops-sys-identity.adb`: Ciało pakietu tożsamości hermetyzujące wywołanie C ABI. Realizuje mechanizm weryfikacji tożsamości (UID 0) bez naruszania czystości domeny wyższych warstw (brak ujawniania specyficznych dla OS typów).
- `source/sys/makeops-sys-processes-os_bindings.ads`: Cienkie, prywatne wiązania do natywnych funkcji jądra Linuksa i `glibc` zarządzających cyklem życia procesów oraz komunikacją IPC (m.in. `fork`, `execvp`, `pipe`, `poll`, `waitpid`).
- `source/sys/makeops-sys-processes.ads`: Specyfikacja bezpiecznej fasady (Thick Wrapper) dla operacji na procesach POSIX. Definiuje w pełnym rygorze SPARK struktury i podprogramy do zarządzania potokami, rozwidlania procesów (`Spawn`), asynchronicznego sprawdzania stanu oraz multipleksowania wejścia/wyjścia.
- `source/sys/makeops-sys-processes.adb`: Implementacja operacji na procesach i strumieniach. Hermetyzuje niebezpieczne wywołania natywnego C ABI (m.in. `fork`, `execvp`, `poll`, `read`), zapewniając obsługę błędów jądra, bezpieczne dekodowanie masek bitowych statusu oraz ochronę przed przepełnieniem buforów (Buffer Overflow).
- `source/sys/makeops-sys-signals.ads`: Specyfikacja bezpiecznej, całkowicie wyizolowanej fasady do asynchronicznego sprawdzania statusu sygnałów przerwań (np. `SIGINT`). Przejmuje wyłączną kontrolę nad sygnałami za pomocą globalnej pragmy konfiguracyjnej `Unreserve_All_Interrupts`.
- `source/sys/makeops-sys-signals.adb`: Implementacja obsługi sygnałów sprzętowych wykorzystująca mechanizm obiektu chronionego (Protected Object). Gwarantuje sprzętową atomowość operacji odczytu/zapisu flagi zakończenia, eliminując ryzyko zjawiska wyścigu (race condition).
- `source/sys/makeops-sys-terminal.ads`: Specyfikacja bezpiecznej, wolnej od wyjątków fasady dla standardowych strumieni terminala (`stdout`, `stderr`). Narzuca rygory SPARK i gwarantuje brak wyjątków w czasie wykonania (AoRE).
- `source/sys/makeops-sys-terminal.adb`: Implementacja fasady terminala. Hermetyzuje natywne wywołania `Ada.Text_IO`, wymusza natychmiastowe opróżnianie buforów (`Flush`) i rygorystycznie tłumi systemowe błędy I/O (np. zamknięte potoki), realizując wzorzec łagodnej degradacji.
- `source/sys/makeops-sys-time.ads`: Specyfikacja bezpiecznego, chronionego rygorem SPARK interfejsu czasu monotonicznego. Definiuje 64-bitowe, odporne na przepełnienia typy pomiaru interwałów i terminów wykonania na potrzeby logiki `Grace Period`.
- `source/sys/makeops-sys-time.adb`: Implementacja adaptująca wbudowany pakiet `Ada.Real_Time` do potrzeb systemu monotonicznego. Izoluje nienależący do domeny SPARK zmienny stan OS (zegar sprzętowy) od deterministycznej, czystej arytmetyki terminów.

- `source/tests/makeops-tests-sys_processes.ads`: Specyfikacja zestawu testów AUnit dla modułu zarządzania cyklem życia procesów systemowych, potoków i sygnałów.
- `source/tests/makeops-tests-sys_processes.adb`: Implementacja kompleksowych testów weryfikujących zachowanie systemu POSIX. Obejmuje pełne ścieżki optymistyczne, głębokie przypadki brzegowe (odporność na `EBADF`, `POLLNVAL`, osierocone deskryptory), symulację zewnętrznych awarii (sygnał `SIGABRT`) oraz stress-testy fizycznych limitów pamięciowych C ABI.
- `source/tests/makeops-tests-sys_signals.ads`: Specyfikacja testu bezpieczeństwa (Sanity Check) dla modułu sygnałów.
- `source/tests/makeops-tests-sys_signals.adb`: Implementacja sprawdzająca deterministyczny stan początkowy flagi przerywań bez wpływania na ciągłość środowiska testowego (odłożenie pełnej symulacji przerwań do testów E2E).
- `source/tests/makeops-tests-sys_terminal.ads`: Specyfikacja zestawu testów AUnit dla bezpiecznej fasady strumieni terminala.
- `source/tests/makeops-tests-sys_terminal.adb`: Implementacja testów pełniących rolę kontroli stabilności (sanity checks). Weryfikuje poprawne wywołania I/O dla ścieżki optymistycznej oraz gwarantuje brak wycieku wyjątków przy przekazywaniu danych brzegowych (puste ciągi znaków, znaki sterujące).
- `source/tests/makeops-tests-sys_time.ads`: Specyfikacja obszernego planu testów jednostkowych (8 scenariuszy) dla zegara monotonicznego.
- `source/tests/makeops-tests-sys_time.adb`: Implementacja rygorystycznie sprawdzająca ścieżki optymistyczne oraz przypadki brzegowe (Edge Cases) pomiaru czasu. Obejmuje stabilność szybkiego odpytywania zegara OS (polling) oraz poprawność arytmetyki z offsetami ujemnymi bez rzucania wyjątków ucięcia (overflow).
- `source/tests/test_runner.adb`: Punkt wejścia dla testów, uruchamiający zestawy za pomocą `AUnit.Run.Test_Runner` i raportujący wyniki w konsoli.
