# ARC-009 MakeOps.App.Config Namespace Architecture

| Attribute | Value |
| :--- | :--- |
| **Document ID** | `ARC-009` |
| **Status** | `APPROVED` |
| **Date** | 2026-05-03 |
| **Target Namespace** | `MakeOps.App.Config` |

## 1. Domain Definition & Purpose

* **Goal:** This domain serves as the central integration and resolution hub for global application preferences. It bridges the gap between static OS environment variables, system-wide configuration files, user-specific overrides, and immediate CLI flags, ensuring that the MakeOps tool behaves predictably across diverse execution contexts (e.g., local developer machines vs. headless CI/CD runners).
* **Core Responsibility:** Manages the Hierarchical Configuration Cascade, systematically querying and merging partial preference structures into a singular, mathematically complete `App_Config` memory record. It operates as the Input Sub-Orchestrator that sets the foundational execution context (such as the Current Working Directory and global Log Level) before the project domain is ever loaded.

## 2. Boundaries & Constraints

* **Domain In-Scope:** Global configuration schema validation, Bottom-Up configuration merging heuristics, `App_Config` and `Partial_Config` target data structures, XDG Base Directory path resolution, and Deferred Observability buffering.
* **Domain Out-of-Scope (Strict Bounds):** This namespace MUST NOT execute operations or perform DAG resolution. It MUST NOT parse raw CLI argument arrays (delegated to `MakeOps.Core.CLI`) or read environment variables directly from the OS (delegated to `MakeOps.Sys.Env` and `MakeOps.Core.Env`). It MUST NOT perform direct TOML lexical parsing, relying strictly on the generic `MakeOps.Core.TOML` frontend pipelines.

## 3. Traceability & Foundation

* **Implements Requirements:**
    * `REQ-004` (Global Tool Preferences):
        * `F-004-006`: Implements the strict 5-level configuration resolution cascade, applying settings in the correct order of precedence (Defaults -> System -> User -> Env -> CLI).
        * `F-004-007`: Enforces graceful degradation by ignoring missing optional configuration files and seamlessly falling back to lower-priority defaults.
        * `NFR-004-003`: Guarantees Separation of Concerns by strictly rejecting preference files containing operational logic (`[operations]` sections).
* **Applies Concepts:**
    * `MOD-003` (Hierarchical Configuration Cascade): Governs the implementation of the Loader, including the Bottom-Up merging heuristic and the concept of partial Monoid-like structures (overlaying `Not_Set` fields).
    * `MOD-019` (Global Preferences Schema Model): Defines the normative structure (`[preferences]`), the security hardening (rejecting `allow_root` in TOML), and the rigid String-Only schema validation performed by the Applier.

## 4. Architectural Topology

**Components (Package Blueprints):**

* `MakeOps.App.Config` (`DES-[XXX]`): The static data repository defining the target memory structures for the application domain. It declares the global `App_Config` record and the `Partial_Config` variant (containing `Not_Set` states) used to store runtime parameters like Log Level and Working Directory. The design MUST be mathematically pure (`pragma Pure`) and rely on strict enumerations to guarantee the absence of invalid states.
* `MakeOps.App.Config.Loader` (`DES-[XXX]`): The Input Sub-Orchestrator responsible for determining global tool preferences. It sequentially instantiates pipelines for the OS Environment and global TOML files (e.g., `/etc/makeops/config.toml`), merging their outputs with flags provided by the CLI using the Bottom-Up heuristic. The design MUST implement the Deferred Observability pattern, buffering non-critical I/O warnings until the final Log Level is resolved, to prevent premature terminal pollution.
* `MakeOps.App.Config.Applier` (`DES-[XXX]`): The Phase 5 semantic Applier dedicated to the application domain. It consumes `Pipeline_Event`s generated by the TOML, CLI, and Env frontends, verifies their semantics against the restricted global schema, and safely populates `Partial_Config` records. The design MUST perform strict validation, immediately halting the pipeline if forbidden execution logic (`[operations]`) is detected in a global scope file.

**Subsystems (Delegated Namespaces):**

* `None` (`N/A`): This namespace constitutes a terminal integration domain for application configuration and contains no further delegated architectural sub-branches.
