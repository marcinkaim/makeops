# MOD-008 System Signal Routing

## 1. Definition & Context
System Signal Routing defines how the MakeOps orchestrator interacts with asynchronous operating system events (such as a user pressing `Ctrl+C`) and how it translates the raw, bit-packed execution status of child processes back into a standard, determinist response for the calling environment.

In the context of the **MakeOps** architecture, handling interruptions safely is vital to prevent "orphaned" child processes from continuously running in the background if the parent orchestrator is killed. Furthermore, rigorous exit code normalization ensures that MakeOps remains fully POSIX-compliant, predictable, and safe to use within automated CI/CD pipelines.

## 2. Theoretical Basis
This model is built upon POSIX process group semantics, hardware interrupt handling theory, and bitmask decoding.

### 2.1. Process Groups and Interrupt Propagation
When an interactive interrupt signal (e.g., `SIGINT` triggered by `Ctrl+C`) or a termination signal (`SIGTERM`) is generated by the terminal, the OS does not send it solely to the parent process. Instead, it broadcasts the signal to the entire Foreground Process Group. Consequently, both MakeOps and the currently running child process (e.g., a compiler) receive the interrupt simultaneously. 
If MakeOps were to die immediately upon receiving this signal, it would abandon the child process and potentially corrupt the terminal state. 

### 2.2. Asynchronous Hardware Interrupts
When the OS delivers a signal, it effectively suspends the main execution thread of the program to invoke a signal handler. In environments bound by strict mathematical verification (SPARK), mutating global state from an unpredictable hardware interrupt introduces severe race conditions and invalidates Absence of Runtime Errors (AoRE) proofs unless mathematically synchronized.

### 2.3. POSIX Exit Status Bitmasks
The POSIX `waitpid` function does not return a simple integer representing the exit code of a process; it returns a packed integer bitmask containing the process's exact fate. This status distinguishes between a process that exited normally (e.g., by calling `exit(1)`) and a process that was terminated by the kernel (e.g., killed by `SIGSEGV` or `SIGKILL`). Extracting this data requires specific bitwise arithmetic (traditionally provided by C macros like `WIFEXITED` and `WEXITSTATUS`).

## 3. Conceptual Model
To safely manage these OS-level constraints, MakeOps abstracts the interrupt logic into a synchronized state machine and enforces a "Grace Period" watchdog.

### 3.1. Protected Object Signal Handler (Concurrency)
To safely intercept OS signals without introducing race conditions, the system leverages Ada's native concurrency primitives (`Ada.Interrupts`). 
A Protected Object acts as an atomic lock mechanism. When the OS interrupt occurs, the hardware temporarily suspends the main thread, executes the protected handler (which simply sets a boolean `Abort_Requested` flag to `True`), and resumes the main thread. The core execution loop (`MOD-005`) safely polls this boolean flag during its standard execution cycle, ensuring zero deadlocks.

### 3.2. The Grace Period Heuristic (Watchdog)
Because the child process receives the `SIGINT` simultaneously with MakeOps, the child *should* terminate itself. However, relying solely on the child introduces the risk of an infinite deadlock if the child explicitly ignores the signal (e.g., via `trap '' INT` in a Bash script) or hangs uncontrollably.
To guarantee termination, the Executor implements a Watchdog heuristic:
1. Upon detecting `Abort_Requested == True`, MakeOps establishes a "Grace Period" deadline timestamp.
2. It continues to poll the child's status non-blockingly.
3. If the child process does not terminate voluntarily before the deadline expires, MakeOps issues a hard `SIGKILL` (Signal 9) directly to the child, forcing the operating system to reap it immediately.

### 3.3. Exit Code Normalization Rule
MakeOps acts as an orchestrator, not a transparent proxy. Regardless of the specific non-zero exit code returned by a failing child process (e.g., `gcc` returning `2`, or a process dying from `SIGSEGV`), MakeOps normalizes the failure. The system aborts the execution queue and returns a unified `Exit_Failure` (POSIX `1`) to the OS environment. This guarantees that the CI/CD pipeline always receives a standard, predictable failure state.

## 4. Engineering Impact

* **Constraints:**
    * All logic related to `Ada.Interrupts` MUST be strictly encapsulated within a dedicated system abstraction package (`MakeOps.Sys.Signals`). It MUST NOT leak into the core orchestration loop to preserve formal SPARK verifiability.
    * The `MakeOps.Sys.Processes.OS_Bindings` package MUST provide Thin Bindings to the standard C macros (`WIFEXITED`, `WEXITSTATUS`, `WIFSIGNALED`, `WTERMSIG`) to correctly decode the `waitpid` bitmask.
    * Any diagnostic output generated by MakeOps during an abort sequence (e.g., "Operation aborted by user") MUST be written exclusively to `stderr` to leave `stdout` unpolluted.
* **Performance Risks:** None. Protected objects responding to hardware interrupts introduce negligible overhead and guarantee thread-safety.
* **Opportunities:** The `Abort_Requested` flag creates a clean integration point. Future enhancements can utilize this flag to perform temporary file cleanups or state rollbacks before exiting the program, providing a highly polished Developer Experience (DX).
