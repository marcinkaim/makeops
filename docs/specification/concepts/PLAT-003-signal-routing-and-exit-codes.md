## 1. Definition & Context
System Signal Routing and Exit Codes dictate how the MakeOps orchestrator interacts with asynchronous operating system events (like a user pressing `Ctrl+C`) and how it reports the final execution status back to the calling environment (such as a CI/CD pipeline).

In the context of the **MakeOps** project, correct signal handling is vital to prevent "orphaned" child processes from running in the background if the parent orchestrator is abruptly killed. Furthermore, rigorous exit code normalization and `stderr` routing ensure that MakeOps remains fully POSIX-compliant, predictable, and safe to use in automated shell scripts where standard output (`stdout`) might be piped to other tools.

## 2. Theoretical Basis
The platform model relies on POSIX process group semantics, bitmask decoding, and Ada's native concurrency primitives.

### 2.1. Process Groups and Interrupt Propagation
When an interactive interrupt signal (e.g., `SIGINT` triggered by `Ctrl+C`) or a termination signal (`SIGTERM`) is generated by the terminal, the OS does not send it solely to the parent MakeOps process. Instead, it broadcasts the signal to the entire Foreground Process Group . Consequently, both MakeOps and the currently running child process (e.g., a compiler) receive the interrupt simultaneously. 
If MakeOps were to die immediately, it would leave the filesystem in an inconsistent state and abandon the child process. To prevent this, MakeOps intercepts the signal, sets an internal atomic `Abort_Requested` flag, and explicitly waits for the child process to gracefully handle its own `SIGINT` and terminate.

**The Grace Period Strategy (Watchdog):**
Relying solely on the child process to handle `SIGINT` introduces the risk of an infinite deadlock if the child explicitly ignores the signal (e.g., via `trap '' INT` in Bash) or hangs uncontrollably. To guarantee termination, MakeOps establishes a "Grace Period" timestamp immediately upon intercepting the user's interrupt. If the child process does not terminate voluntarily before this deadline expires, MakeOps issues a hard `SIGKILL` (Signal 9) directly to the child, forcing the operating system to reap it.

### 2.2. Exit Status Bitmask Decoding
The POSIX `waitpid` function (invoked asynchronously via the `WNOHANG` flag during the multiplexing loop) does not return a simple integer representing the exit code; it returns a packed bitmask containing the process's fate.
To accurately report what happened to a child process, MakeOps utilizes Thin Bindings to standard C macros:
* `WIFEXITED(status)`: Evaluates to true if the child terminated normally.
* `WEXITSTATUS(status)`: Extracts the actual 8-bit exit code (0-255).
* `WIFSIGNALED(status)`: Evaluates to true if the child was killed by an unhandled signal (e.g., `SIGSEGV` or `SIGKILL`).
* `WTERMSIG(status)`: Extracts the number of the fatal signal.

### 2.3. Asynchronous Signal Handling via `Ada.Interrupts`
To safely intercept OS signals without introducing race conditions, the system leverages Ada's `Ada.Interrupts` package. A Protected Object is defined with a procedure attached to specific OS interrupts. When the interrupt occurs, the hardware temporarily suspends the main thread, executes the protected handler (which simply toggles the boolean `Abort_Requested` flag), and resumes the main thread. This avoids the severe restrictions typically associated with unsafe POSIX signal handlers in C.

## 3. Engineering Impact

* **Constraints:**
    1. **Unified Exit Strategy:** MakeOps acts as an orchestrator, not a transparent proxy. Regardless of the specific non-zero exit code returned by a failing child process (e.g., `gcc` returning `2`), MakeOps MUST abort the execution queue and return a unified `Exit_Failure` (typically `1`) to the OS environment.
    2. **Strict Stream Routing:** Any diagnostic output generated by MakeOps itself (e.g., "Operation aborted by user" or syntax errors) MUST be written exclusively to `stderr` via the safe `MakeOps.Sys.Terminal` facade, leaving `stdout` strictly reserved for the output of child processes.
    3. **Concurrency Isolation:** All logic related to `Ada.Interrupts` and the protected object managing the signal state MUST be strictly encapsulated within a dedicated system abstraction package (`MakeOps.Sys.Signals`). It MUST NOT leak into the core orchestration loop to preserve formal verifiability.
* **Performance Risks:** None. Protected objects responding to hardware interrupts introduce negligible overhead and guarantee thread-safety.
* **Opportunities:** The `Abort_Requested` flag allows the internal DAG traversal loop to perform cleanups (e.g., deleting temporary files generated by MakeOps) before exiting the program cleanly, providing a highly polished Developer Experience (DX).
